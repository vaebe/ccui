# 测试原则（术语 + 优先级）

> 状态：**已采纳，作为长期不变性约束**
> 适用范围：写新测试、评审测试 PR、给 AI / Codex 下测试任务、生成覆盖率讨论

ccui 是 Vue 组件库。测试是组件「外部契约」的可执行规格——所以测试用的语言、断言的对象，都要跟用户实际写 Vue 代码时的心智对齐。

## 一、Vue 状态语义

写测试、demo、评审说明、issue 描述时，优先使用 Vue 语境的词：

| 场景 | 用 | 不用 |
| --- | --- | --- |
| 父组件维护的状态 | `v-model:*` / `update:*` / `props 驱动` / `外部状态接管` | `controlled` |
| 组件内部维护的状态 | `default*` / `内部状态` / `组件自行维护` | `uncontrolled` |

可以在内部讨论中以「类似 controlled / uncontrolled 的状态模式」辅助理解，但落到文档、测试命名、错误信息时，写 Vue 用户实际会用到的 API 名。

跟 [`benchmark-principles.md`](./benchmark-principles.md) 的「不照搬 React-only 模式」一致——那条约束面向实现，本条约束面向测试与文档表达。

## 二、测试优先级

不要为了覆盖率数字而测。新增测试应优先证明以下真实行为：

- 用户可感知的渲染、交互、键盘、滚动、拖拽、定时器行为
- `v-model:*` 与 `update:*` 事件协议
- `default*` 初始值与组件内部状态变化
- `disabled` / `readonly` / `loading` / `empty` / `error` / `fallback` 等边界状态
- slot、自定义渲染、自定义 class / icon / style 等公开 API
- 组件卸载、容器清理、事件监听清理等容易回归的副作用

**不建议**为了覆盖率硬测：

- 私有实现细节
- 只为命中行数的 Transition hook
- 类型文件默认值
- 对用户行为没有证明价值、且会让测试变脆弱的 DOM 细节

覆盖率是结果不是目标。具体数字基线随版本漂移，需要时跑 `pnpm --filter @vaebe/ccui test:coverage`，不在本文件冻结。

## 反向决策开销

- 是测试与表达层级的不变性约束。单个测试中偶然用词偏离可以个别修正，不需要全局回滚。
- 真要废弃整套原则，等价于把 ccui 按 React 受控模型来测，那是另一个项目。
